Executable

An Executable (pushtopics__Executable__c) is a sophisticated configuration object that orchestrates object-level operations and provides a wide range of data management capabilities.

  • Extraction of data from the source, subsequent transformation, and then performing actions on the target.
  • Retrieval of filtered data from any Salesforce org for flexible display on lightning pages.
  • Data upload via CSV files.

Key configurations encapsulated by the Executable are:

  • Source Data Retrieval: Utilizes Salesforce SOQL to dictate data extraction methods.
  • Target Data Existence Check: Includes checks for target existence via the Target Matching Field.
  • Target Data Action: Enables actions like Insert, Update, Upsert, Delete with detailed control.
  • Association with Field Mappings: Facilitates advanced field-level transformations through associated Field Mapping records.
  • Scope Filter: Includes in-memory checks to verify if a source record is within the defined execution scope.
  • Delta Load: Ensures retrieval of data that has changed since the last execution.
  • Execution Order: Dictates the sequence of operations during job executions.
  • Batch Size: Defines the quantity of data processed during each batch operation.
  • Logging and Notifications: Provides guidelines for tracking system activities and sending notifications.
  • Serial Mode Execution: Specifies if operations should run in sequence.
  • Trigger Execution: Enables transformation during trigger execution.
  • "Q" component settings: Offers granular control over the "Q" component settings, ensuring smooth integration with Lightning features.

In summary, an Executable is a pivotal component of the DSP system, shaping the configuration for object-level operations, aligning with field-level mappings, and providing dynamic data management capabilities through the control of its "Q" component.

Fields

NameDeveloper NameDescription
Actionpushtopics__Action__cThe Action DSP executes against the Target. There are 4 options:
  • Insert: only performs the 'Insert' action against the Target. If the source data already exists in the Target, the existing data will be ignored for this Action.
  • Update: only performs 'Update' action against the Target. Data that does not exist in the Target will be ignored and only 'Update' is performed.
  • Upsert: default option. DSP performs the 'Insert' action against the Target for non-existant data, and the 'Update' action on existing data.
  • Delete: DSP performs 'Delete' action if the record exists in the target.
    All or Nothing?pushtopics__AllOrNothing__cDefault is 'false'. Any record failures will cause the entire transaction roll back. This can only be checked when 'Batchable' is unchecked and the Target is an integration type of Connection.
    Always Transform When Records Updated? pushtopics__AlwaysTransformWhenUpdated__cUsed along with the 'Transform in Before Triggers?' checkbox in before update triggers. If checked, DSP will run the transformation whenever the records are updated; otherwise, the transformation will only execute when the related data's change triggered the update to the current object.
    Batch Sizepushtopics__BatchSize__cMax value is 200. Once data is retrieved from Source, the data will be split into Action batches based on the value of this field. For example: It may be a quick process to retrieve 400 records from a Source, but when actioning against the Target, due to heavy triggers/workflows/process builders which can impact the DML time, it could be better to perform the actions 50 records a time. So the 'Retrieve Size' can be set to 400, and 'Action Batch Size' be 50. This option makes it flexible to adjust the size of the batches in execution separately.
    Batchable?pushtopics__Batchable__cDefault is 'true'. If checked, the Execution will run in batch mode, where each batch ends up with a Batch Execution record created, which holds the details of a batch execution, and the Execution record holds the summarized values. If unchecked, all the details are logged in the Execution record and no Batch Execution records will be created. The non-batch mode helps reduce the layer of logs, but is only supposed to be used when the dataset is small enough to be handled within a single APEX transaction, without hitting Salesforce governor limits. 'Batchable?' and 'All or Nothing?' cannot both be checked.
    Bypass Duplicate Rule Alerts?pushtopics__BypassDuplicateRuleAlerts__cDefault is 'false'. If checked, and if the data actioned to the Target triggers the Duplicate Rule Alert, the alert will be bypassed.

    Note:

    A System Administrator will bypass the alert regardlessly whether it is checked or not.
